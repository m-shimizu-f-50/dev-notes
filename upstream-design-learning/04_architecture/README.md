# 🏛️ Phase 4：アーキテクチャ設計

> **「システム全体の骨格・構造」を設計するフェーズ**
> 技術選定・構成図・非機能要件への対応を決めます。

---

## アーキテクチャ設計とは？

アーキテクチャ設計とは、**「システムをどんな構造・構成で作るか」**を決める設計です。家の建築に例えると、「木造か鉄骨か」「平屋か二階建てか」という骨格を決める作業です。

良いアーキテクチャは：
- 将来の変更・拡張がしやすい
- 障害に強い（1か所壊れても全体が止まらない）
- 性能要件を満たせる

---

## 代表的なアーキテクチャパターン

### 1. モノリシックアーキテクチャ

全ての機能が1つのアプリケーションに含まれる構成。

```
┌────────────────────────────────────┐
│      Webアプリケーション（1つ）      │
│  ┌────────┐ ┌────────┐ ┌────────┐  │
│  │ 認証機能 │ │ 商品機能 │ │ 注文機能 │  │
│  └────────┘ └────────┘ └────────┘  │
└────────────────────────────────────┘
                    │
              ┌─────┴─────┐
              │  データベース │
              └───────────┘
```

**メリット：**
- シンプルで開発・デプロイが楽
- 小規模システムに向いている

**デメリット：**
- 規模が大きくなると複雑化する
- 一部の変更が全体に影響する

---

### 2. レイヤードアーキテクチャ（3層構成）

最も一般的な構成。役割ごとに層（レイヤー）に分ける。

```
【クライアント】
  ブラウザ / スマートフォンアプリ
       │
       │ HTTPS
       ↓
【プレゼンテーション層（フロントエンド）】
  画面表示・ユーザー操作
       │
       │ API通信
       ↓
【アプリケーション層（バックエンド）】
  ビジネスロジック・API処理
       │
       │ SQL
       ↓
【データ層（DB）】
  データの永続化・管理
```

---

### 3. マイクロサービスアーキテクチャ

機能ごとに独立したサービス（アプリ）に分ける構成。大規模システム向け。

```
┌─────────┐  ┌─────────┐  ┌─────────┐
│  認証    │  │  商品    │  │  注文    │
│ サービス │  │ サービス │  │ サービス │
└────┬────┘  └────┬────┘  └────┬────┘
     │             │             │
     └─────────────┼─────────────┘
                   │
            ┌──────┴──────┐
            │  APIゲートウェイ │
            └──────┬──────┘
                   │
            ┌──────┴──────┐
            │   クライアント  │
            └─────────────┘
```

**メリット：**
- 各サービスを独立してデプロイ可能
- スケールアップが柔軟

**デメリット：**
- システムが複雑になる
- 運用コストが上がる

## 非機能要件への対応設計

### 性能（パフォーマンス）

```
主な対策：
1. キャッシュの活用
   - Redisなどのキャッシュサーバーで頻繁なDB問い合わせを削減

2. CDNの利用
   - 静的ファイル（画像・CSS・JS）をCDNで配信

3. DBのインデックス最適化
   - よく検索するカラムにインデックスを設定

4. ロードバランサー
   - 複数サーバーに負荷を分散
```

### 可用性（ダウンしない）

```
主な対策：
1. 冗長化
   - サーバーを複数台用意（1台落ちても動き続ける）

2. マルチAZ構成
   - クラウドでは異なるデータセンターに配置

3. ヘルスチェック
   - 自動的に死活監視し、異常サーバーを切り離す

4. データバックアップ
   - DBを定期的にバックアップ
```

### セキュリティ

```
主な対策：
1. 通信の暗号化
   - HTTPS（TLS）を必ず使用

2. 認証・認可
   - 適切なログイン・アクセス制御

3. ファイアウォール
   - 不要なポートへのアクセスをブロック

4. WAF（Webアプリケーションファイアウォール）
   - SQLインジェクション・XSSなどへの防御
```

## 技術選定の考え方

アーキテクチャ設計では、使用する技術を選定します。

| 観点 | 考慮すること |
|------|------------|
| チームのスキル | 使い慣れた技術を選ぶ（学習コスト） |
| 要件への適合 | リアルタイム性が必要？大量データ？ |
| スケーラビリティ | 将来の拡張に対応できるか |
| コスト | ライセンス費用・インフラ費用 |
| コミュニティ | 情報が豊富か、サポートがあるか |
| 保守性 | 長期的にメンテナンスできるか |

## アーキテクチャ設計書に含める内容

1. **システム構成図**：全体の構成を図で表す
2. **技術スタック**：使用する言語・FW・インフラ
3. **非機能要件への対応方針**
4. **データフロー**：データがどう流れるか
5. **セキュリティ方針**
6. **障害対応方針**（バックアップ・復旧手順）

## 設計原則（押さえておくべき考え方）

### SOLID原則（オブジェクト指向設計）

| 原則 | 意味 |
|------|------|
| S - Single Responsibility | 1クラスは1つの責務だけ持つ |
| O - Open/Closed | 拡張には開いて、修正には閉じる |
| L - Liskov Substitution | サブクラスは親クラスと置き換え可能 |
| I - Interface Segregation | 使わないインターフェースを強要しない |
| D - Dependency Inversion | 抽象に依存し、具体に依存しない |

### 疎結合・高凝集

```
疎結合：各部品はなるべく独立していること
  → 変更の影響範囲が小さくなる

高凝集：関連する機能は1か所にまとめること
  → 見つけやすく、変更しやすくなる
```
