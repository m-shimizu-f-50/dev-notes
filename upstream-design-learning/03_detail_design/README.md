# ⚙️ Phase 3：詳細設計（内部設計）

> **「システムの内部がどう動くか」を設計するフェーズ**
> 実装者が迷わずコードを書けるレベルまで落とし込みます。

---

## 詳細設計とは？

基本設計が「外から見た設計」なら、詳細設計は**「内部の動きを細かく定義する設計」**です。

```
基本設計　→　「商品を検索できる画面を作る」
     ↓
詳細設計　→　「検索ボタン押下時、キーワードをINPUTとして
             productsテーブルのnameカラムをLIKE検索し、
             結果をproduct_id順で最大50件返す」
```

詳細設計が不十分だと、実装者によって実装が変わってしまいます。

---

## 詳細設計に含まれる内容

1. **テーブル定義（物理設計）**
2. **処理フロー設計（フローチャート）**
3. **API設計**
4. **クラス・モジュール設計**
5. **シーケンス図**

---

## 1. テーブル定義（物理設計）

基本設計で作ったER図を元に、実際のテーブル・カラムを定義します。

### テーブル定義書の例

**usersテーブル**

| カラム名 | データ型 | NULL | デフォルト | 説明 |
|---------|---------|------|----------|------|
| id | BIGINT | NOT NULL | AUTO_INCREMENT | ユーザーID（主キー） |
| name | VARCHAR(100) | NOT NULL | — | ユーザー名 |
| email | VARCHAR(255) | NOT NULL | — | メールアドレス（一意） |
| password_hash | VARCHAR(255) | NOT NULL | — | パスワード（ハッシュ化） |
| created_at | DATETIME | NOT NULL | CURRENT_TIMESTAMP | 作成日時 |
| updated_at | DATETIME | NOT NULL | CURRENT_TIMESTAMP | 更新日時 |
| deleted_at | DATETIME | NULL | NULL | 削除日時（論理削除） |

**ポイント：**
- `id`はBIGINTを使う（将来のデータ増加に備えて）
- パスワードは必ずハッシュ化（平文保存は絶対NG）
- `deleted_at`で論理削除（実データは残す）
- `created_at` / `updated_at`は必ず持つ

## 2. 処理フロー設計（フローチャート）

処理の流れを図で表します。特にロジックが複雑な部分は必ず書きます。

```
【フローチャートの例：ログイン処理】

  開始
    │
    ↓
  メール・パスワードを受け取る
    │
    ↓
  DBからメールアドレスでユーザーを検索
    │
    ↓
  ユーザーが存在するか？
    ├── NO → エラー：「認証失敗」
    │
    └── YES ↓
            パスワードが一致するか？
              ├── NO → エラー：「認証失敗」
              │
              └── YES ↓
                      アカウントが有効か？
                        ├── NO → エラー：「アカウントが無効」
                        │
                        └── YES ↓
                                セッションを発行
                                  │
                                  ↓
                                TOP画面へリダイレクト
                                  │
                                  ↓
                                終了
```

### フローチャートの記号

| 記号 | 意味 |
|------|------|
| 楕円 | 開始・終了 |
| 四角 | 処理 |
| ひし形 | 条件分岐 |
| 平行四辺形 | 入出力 |
| 矢印 | 処理の流れ |

## 3. API設計

フロントエンドとバックエンド、またはシステム間の通信仕様を定義します。

### API仕様書の例（REST API）

**商品一覧取得 API**

```
GET /api/v1/products

リクエスト：
  クエリパラメータ：
    - keyword: string（任意）　検索キーワード
    - category_id: number（任意）　カテゴリID
    - page: number（任意・デフォルト1）　ページ番号
    - per_page: number（任意・デフォルト20）　1ページの件数

レスポンス（200 OK）：
{
  "data": [
    {
      "id": 1,
      "name": "サンプル商品",
      "price": 1000,
      "image_url": "https://example.com/images/1.jpg",
      "category": {
        "id": 3,
        "name": "家電"
      }
    }
  ],
  "pagination": {
    "total": 150,
    "page": 1,
    "per_page": 20,
    "last_page": 8
  }
}

エラーレスポンス（400 Bad Request）：
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "per_pageは100以下で指定してください"
  }
}
```

### HTTPステータスコードの使い方

| コード | 意味 | 使用場面 |
|--------|------|---------|
| 200 | 成功 | GET・PUT・PATCH成功時 |
| 201 | 作成成功 | POST（新規作成）成功時 |
| 204 | コンテンツなし | DELETE成功時 |
| 400 | 不正リクエスト | バリデーションエラー |
| 401 | 未認証 | ログインが必要な場合 |
| 403 | 権限なし | アクセス権がない場合 |
| 404 | 見つからない | リソースが存在しない |
| 500 | サーバーエラー | 予期しないエラー |

## 詳細設計でよく出てくる概念

### 正規化

データの重複をなくし、一貫性を保つためのDB設計手法。

```
❌ 正規化されていないテーブル
orders テーブル:
  order_id | user_name | user_email | product_name | price
  1        | 田中太郎  | taro@...   | 商品A        | 1000
  2        | 田中太郎  | taro@...   | 商品B        | 2000
  ※ ユーザー情報が重複している → メアド変更時に全行修正が必要

✅ 正規化されたテーブル
users: id | name  | email
       1  | 田中太郎 | taro@...

orders: id | user_id | product_id | quantity
        1  | 1       | 101        | 1
```

### インデックス設計

検索を高速化するためのDB設定。よく検索するカラムに設定します。

```sql
-- メールアドレスで検索することが多いため
CREATE INDEX idx_users_email ON users(email);

-- 注文のuser_idで絞り込むことが多いため
CREATE INDEX idx_orders_user_id ON orders(user_id);
```
