# 01. 基礎概念：RDBMSとテーブル設計の全体像

---

## 1. RDBMSとは何か

**RDBMS（Relational Database Management System）** とは、データを「表（テーブル）」の形で管理するデータベースシステム。

代表的なもの：
- **MySQL / MariaDB** → Webシステムで最も普及
- **PostgreSQL** → 機能が豊富、最近のモダンな開発でよく使われる
- **SQLite** → 組み込み・開発環境向け
- **Oracle / SQL Server** → 大企業向けエンタープライズ

「リレーショナル」= テーブルどうしを**関係（リレーション）**でつなぐという意味。

---

## 2. テーブルの基本構造

```
usersテーブル

| id | name   | email              | created_at          |
|----|--------|--------------------|---------------------|
|  1 | 田中   | tanaka@example.com | 2024-01-01 10:00:00 |
|  2 | 佐藤   | sato@example.com   | 2024-01-02 09:30:00 |
```

- **テーブル（Table）**：データを格納する表。エンティティ（概念）ごとに作る
- **カラム（Column）**：列。どんな属性を持つかを定義する（名前・型・制約）
- **レコード / 行（Row）**：1件のデータ
- **セル**：1カラム×1レコードの交点。1つの値が入る

---

## 3. テーブル設計とは何をすることか

テーブル設計とは「**どんなテーブルを、どんな構造で作るか**」を決めること。

具体的には：

1. **どのテーブルが必要か**（エンティティの洗い出し）
2. **各テーブルにどのカラムが必要か**（属性の定義）
3. **カラムのデータ型は何か**（INT / VARCHAR / TIMESTAMP など）
4. **テーブル間の関係をどうするか**（外部キー・中間テーブルなど）
5. **どこにインデックスを貼るか**（検索パフォーマンス）
6. **どんな制約をかけるか**（NOT NULL / UNIQUE / CHECK など）

これらすべてを「**ER図（Entity-Relationship Diagram）**」として図示することが多い。

---

## 4. ER図の読み方

```
users                        orders
┌──────────────┐             ┌──────────────────┐
│ id (PK)      │──────────┐  │ id (PK)          │
│ name         │          └─>│ user_id (FK)     │
│ email        │             │ total_amount     │
│ created_at   │             │ ordered_at       │
└──────────────┘             └──────────────────┘
```

- **PK（Primary Key）**：主キー。そのテーブルで行を一意に識別する
- **FK（Foreign Key）**：外部キー。他テーブルのPKを参照する
- **線の種類**：1対1、1対多、多対多などの関係を表す

---

## 5. テーブル設計の全体フロー

```
要件定義
   ↓
エンティティの洗い出し     ← 「何を管理したいか」を列挙する
   ↓
属性の定義                 ← 各エンティティが持つ情報を決める
   ↓
テーブル設計（論理設計）   ← 正規化・リレーション・制約を決める
   ↓
物理設計                   ← データ型・インデックス・パーティションなど
   ↓
DDL作成（CREATE TABLE）    ← 実際のSQL
```

---

## 6. エンティティの洗い出し方

例：ECサイトを作るとしたら

**「誰が・何を・いつ・どのように」を考える**

| 誰が | 何を | どのように |
|------|------|-----------|
| ユーザー | 商品を | 注文する |
| 管理者 | 商品を | 登録・更新する |
| ユーザー | 商品に | レビューを書く |
| ユーザー | 住所を | 管理する |

→ **テーブル候補**：users, products, orders, order_items, reviews, addresses, admins

**コツ：名詞がテーブルになる、動詞がリレーションになる**

---

## 7. 「良い設計」の基準

実務で求められる設計の基準：

| 基準 | 説明 |
|------|------|
| **データの整合性** | 矛盾したデータが入らない構造になっているか |
| **重複がない** | 同じデータが複数箇所に散らばっていないか |
| **拡張しやすい** | 後から仕様変更が来ても影響が少ない構造か |
| **パフォーマンス** | よく使うクエリが効率よく動くか |
| **読みやすい** | 他の開発者がER図やDDLを見てすぐ理解できるか |

---

## 8. まとめ

- RDBMSはテーブルでデータを管理し、テーブル間をリレーションでつなぐ
- テーブル設計とは「何を・どんな構造で・どんな関係で格納するか」を決めること
- 要件から「名詞（エンティティ）」を洗い出し、属性を定義するのが出発点
- 良い設計は「整合性・非重複・拡張性・パフォーマンス・可読性」のバランスで判断する
