# 📋 Phase 1：要件定義

> **「何を作るか」を正しく決めることが、プロジェクト成功の9割を決める**

---

## 要件定義とは？

要件定義とは、**システムで「何をしたいか」を明確にするフェーズ**です。

例えるなら、家を建てる前に「何部屋欲しいか」「どんな間取りにしたいか」を決める作業です。ここでズレが生じると、後の工程でどれだけ頑張っても「思っていたものと違う」ものができてしまいます。

```
よくある失敗パターン ❌
  クライアント「検索機能が欲しいです」
  エンジニア  「わかりました！全文検索で実装します」
  　↓ リリース後
  クライアント「え？部分一致だけでよかったのに…コスト高すぎです」

要件定義がしっかりしていれば防げた問題！
```

---

## 要件の種類

### 機能要件

**「システムが何をするか」**

- ユーザーがログインできる
- 商品を検索できる
- 注文内容をメールで通知する

### 非機能要件

**「システムがどのくらいの品質で動くか」**

| 項目             | 例                                           |
| ---------------- | -------------------------------------------- |
| 性能             | 1秒以内にレスポンスを返す                    |
| 可用性           | 24時間365日稼働（ダウンタイム年間1時間以内） |
| セキュリティ     | 個人情報は暗号化して保存する                 |
| 保守性           | 機能追加が容易な構造にする                   |
| スケーラビリティ | ユーザー数が10倍になっても対応できる         |

※ダウンタイムとはシステムやサービスが停止して使えない時間 のこと
※スケーラビリティとは利用者やデータ量が増えても、問題なく処理できる能力 のこと

> ⚠️ **非機能要件を見落とすな！**
> 機能要件だけ決めて設計を進めると、性能やセキュリティの問題が実装後に発覚しがちです。

## 要件定義の進め方

### ステップ1｜ヒアリング

まず、クライアント・ユーザーから情報を引き出します。

**ヒアリングで押さえるポイント：**

```
5W1H で考える
  - Who（誰が使うのか？）　→ システムのユーザー像
  - What（何をしたいのか？）→ 達成したいゴール
  - When（いつ使うのか？）　→ 利用シーン・頻度
  - Where（どこで使うのか？）→ 環境（PC？スマホ？）
  - Why（なぜ必要なのか？）　→ 課題・背景
  - How（どうやって使うのか？）→ 操作フロー
```

**ヒアリングのコツ：**

- 「YesかNoで答えられる質問」より「オープンな質問」をする
  - ❌「検索機能は必要ですか？」
  - ✅「商品を探す時、どういう方法で探したいですか？」
- 「あたりまえ品質」は言葉にならないので、自分から確認する
- 曖昧な言葉（「使いやすく」「速く」「たくさん」）は必ず数値化する

---

### ステップ2｜ユーザーストーリーの作成

**ユーザーストーリー**とは、ユーザー視点で機能を記述する手法です。

```
テンプレート：
「[ユーザーの種類] として、[したいこと] がしたい。
 なぜなら、[理由・目的] だからだ。」

例：
「会員ユーザー として、過去の注文履歴を確認 したい。
 なぜなら、再注文をスムーズにしたい からだ。」
```

---

### ステップ3｜ユースケース図

ユーザーとシステムの関係を図で表します。

```
【ユースケース図の例：ECサイト】

  ┌────────────────────────────────────────────┐
  │               ECサイト                     │
  │                                            │
  │  ┌──────────────┐  ┌──────────────┐       │
  │  │  商品を検索   │  │  注文をする   │       │
  │  └──────────────┘  └──────────────┘       │
  │                                            │
  │  ┌──────────────┐                          │
  │  │ レビューを書く│                          │
  │  └──────────────┘                          │
  │                                            │
  └────────────────────────────────────────────┘
          ↑                    ↑
     一般ユーザー            会員ユーザー
```

---

### ステップ4｜要件定義書の作成

ヒアリング内容を文書化します。

**要件定義書に含める内容：**

1. **プロジェクト概要**：目的・背景・ゴール
2. **システム概要**：対象システムの全体像
3. **ユーザー定義**：誰が使うか（ペルソナ）
4. **機能要件一覧**：機能のリスト
5. **非機能要件**：性能・可用性・セキュリティ等
6. **制約条件**：予算・納期・技術的制約
7. **用語定義**：プロジェクト内で使う言葉の統一

---

## 要件定義書のサンプル構成

```markdown
# 要件定義書

## 1. プロジェクト概要

- 目的：〇〇の課題を解決するため
- 背景：現状〇〇という問題がある
- ゴール：〇〇を実現する

## 2. ユーザー定義

| ユーザー種別 | 説明                         |
| ------------ | ---------------------------- |
| 一般ユーザー | 商品を購入するお客様         |
| 管理者       | 商品・在庫を管理するスタッフ |

## 3. 機能要件

| ID   | 機能名   | 説明                           | 優先度 |
| ---- | -------- | ------------------------------ | ------ |
| F001 | ログイン | メール+パスワードでログイン    | 必須   |
| F002 | 商品検索 | キーワード・カテゴリで絞り込み | 必須   |

## 4. 非機能要件

| 項目       | 要件                |
| ---------- | ------------------- |
| レスポンス | 通常操作は2秒以内   |
| 可用性     | 年間稼働率99.9%以上 |
```

## よくある失敗と対策

| 失敗パターン               | 原因               | 対策                               |
| -------------------------- | ------------------ | ---------------------------------- |
| 「なんとなく決めた」要件   | ヒアリング不足     | 5W1Hで深掘りする                   |
| 後から要件が増える         | スコープ未定義     | 最初にスコープを明確に合意する     |
| 関係者間で認識がズレる     | 口頭だけで進める   | 必ず文書化・レビューする           |
| 「できます」と言ってしまう | 実現可能性の未確認 | 技術的な実現可能性を事前に確認する |
