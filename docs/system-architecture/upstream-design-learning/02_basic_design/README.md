# 🏗️ Phase 2：基本設計（外部設計）

> **「システムを外から見たときの姿」を決めるフェーズ**
> ユーザーやクライアントが「見える・触れる」部分の設計を行います。

---

## 基本設計とは？

要件定義で「何を作るか」が決まったら、次は「どんな見た目・構造にするか」を決めます。これが**基本設計（外部設計）**です。

```
要件定義　→　「ユーザーが商品を検索できること」
     ↓
基本設計　→　「検索画面の項目・レイアウト」「検索結果の表示形式」
             「どのシステムと連携するか」
     ↓
詳細設計　→　「DBのクエリはどう書くか」（次のフェーズ）
```

---

## 基本設計に含まれる内容

1. **画面設計**（UI設計）
2. **画面遷移設計**
3. **データベース概念設計**（ER図）
4. **外部インターフェース設計**（他システムとの連携）
5. **バッチ・帳票設計**（必要に応じて）

---

## 1. 画面設計（UI設計）

### ワイヤーフレームとは？

画面の骨格を「枠組みだけ」で描いたものです。デザインではなく「配置と機能」を確認するために使います。

```
【ワイヤーフレームの例：ログイン画面】

  ┌─────────────────────────────────┐
  │           ロゴ                  │
  │                                 │
  │  メールアドレス                  │
  │  ┌─────────────────────────┐   │
  │  │ example@mail.com        │   │
  │  └─────────────────────────┘   │
  │                                 │
  │  パスワード                      │
  │  ┌─────────────────────────┐   │
  │  │ ••••••••                │   │
  │  └─────────────────────────┘   │
  │                                 │
  │  ┌─────────────────────────┐   │
  │  │      ログイン           │   │
  │  └─────────────────────────┘   │
  │                                 │
  │  パスワードを忘れた方はこちら     │
  └─────────────────────────────────┘
```

### 画面設計書に書くこと

各画面について以下を定義します：

| 項目 | 内容 |
|------|------|
| 画面ID | 一意の識別子（例：SCR-001） |
| 画面名 | ログイン画面 |
| 表示条件 | 未ログイン時のみ表示 |
| 入力項目 | メールアドレス（必須・形式チェック）、パスワード（必須・8文字以上） |
| ボタン | ログインボタン → ログイン処理実行 |
| エラー表示 | 認証失敗時：「メールアドレスまたはパスワードが正しくありません」 |

画面と画面のつながりを図で表します。

```
【画面遷移図の例：ECサイト】

  ┌────────────┐    ログイン成功    ┌────────────┐
  │ ログイン画面 │ ────────────── → │  TOP画面   │
  └────────────┘                  └─────┬──────┘
                                        │ 商品クリック
                                        ↓
                                  ┌────────────┐
                                  │ 商品詳細画面 │
                                  └─────┬──────┘
                                        │ カートに追加
                                        ↓
                                  ┌────────────┐
                                  │  カート画面  │
                                  └─────┬──────┘
                                        │ 購入手続きへ
                                        ↓
                                  ┌────────────┐
                                  │  注文確認画面 │
                                  └────────────┘
```

## 3. データベース概念設計（ER図）

### ER図（Entity-Relationship図）とは？

データの「もの（エンティティ）」と「関係（リレーション）」を図で表したものです。

```
【ER図の例：ECサイト】

  ┌─────────┐          ┌─────────┐
  │  ユーザー │ 1 ──── n │   注文   │
  └─────────┘          └────┬────┘
                             │ 1
                             │
                             │ n
                        ┌────┴────┐          ┌─────────┐
                        │ 注文明細 │ n ──── 1 │   商品   │
                        └─────────┘          └─────────┘
```

### エンティティとは？

データとして管理する「もの」のことです。

```
例：ECサイトのエンティティ
  - ユーザー（user_id, name, email, ...）
  - 商品（product_id, name, price, stock, ...）
  - 注文（order_id, user_id, order_date, ...）
  - 注文明細（detail_id, order_id, product_id, quantity, ...）
```

### リレーション（関係）の種類

| 関係 | 意味 | 例 |
|------|------|-----|
| 1対1 | 一方が一方に1つだけ対応 | ユーザー ↔ プロフィール |
| 1対多 | 一方が複数に対応 | ユーザー → 注文（1人が複数注文） |
| 多対多 | 双方が複数に対応 | 商品 ↔ タグ（1商品に複数タグ、1タグが複数商品に付く） |

## 4. 外部インターフェース設計

他のシステムやサービスとどう連携するかを定義します。

```
【外部連携の例：ECサイト】

  自社ECシステム
      │
      ├── 決済サービス（クレジットカード決済）
      ├── 配送業者システム（配送状況の連携）
      ├── メール配信サービス（注文完了メール）
      └── 在庫管理システム（在庫同期）
```

**インターフェース設計書に書くこと：**

| 項目 | 内容 |
|------|------|
| 連携先 | 決済サービスAPI |
| 通信方式 | REST API（HTTPS） |
| データ形式 | JSON |
| タイミング | 注文確定時（リアルタイム） |
| エラー時の動作 | 3回リトライ後、エラー画面表示 |

---

## 基本設計書のレビューポイント

設計書が完成したらレビューで以下を確認します：

- [ ] 要件定義の全機能が設計に反映されているか
- [ ] ユーザーが操作しやすい画面設計になっているか
- [ ] エラーケースが設計されているか
- [ ] データの整合性が保たれる設計になっているか
- [ ] 他システムとの連携が明確か

## 実際にやってみよう！

**ミニ演習：自分でやってみる**

```
お題：「シンプルなTODOアプリ」の基本設計をやってみよう

手順：
1. 画面を洗い出す（TOP画面、タスク追加画面、など）
2. 画面遷移図を紙に書く
3. データを洗い出す（タスク名・期限・完了フラグ、など）
4. ER図を書く

これだけで基本設計の流れが体験できます！
```
